# VaultLedger — Product Spec

**Working name:** VaultLedger
**One-liner:** Git for agent beliefs — a governed memory broker that lets any AI agent use an Obsidian vault (or any markdown folder) as persistent memory, with provenance, approval, and rollback enforced in code, not prompts.
**Stack:** TypeScript throughout (Node MCP server + write broker; Obsidian plugin; SQLite index)
**Status:** Spec v1.1 — July 3, 2026 (v1.1 folds in the Undertow lifecycle merge; see `docs/design/specs/2026-07-03-undertow-integration-decision.md`)

---

## 1. Problem

AI agents (Claude Code, Codex, Claudian, scheduled vault agents) can now read and write Obsidian vaults as memory. But every existing solution governs writes with *prompts* — instructions the model can ignore, hallucinate around, or violate under drift. There is no enforcement layer. Users either lock agents out of their real notes (memory stays shallow) or let them in and hope (knowledge gets silently corrupted). The 2026 agent-memory market (Mem0, Zep, Letta) has the same hole: no lineage, no audit, no rollback.

## 2. Solution

A local service with three parts:

1. **MCP memory server** — exposes memory tools (`remember`, `recall`, `revise`, `promote`, `forget`) to any MCP client. Agents never touch files directly.
2. **Deterministic write broker** — validates every proposed operation against a permissions manifest, applies patch-level edits with hash checks, stamps provenance, commits each transaction to Git, and queues protected-zone writes for approval.
3. **Obsidian review plugin** — the "what does my agent believe" surface: approval queue with diffs, provenance on hover, one-click rollback, staleness and contradiction queues.

## 3. Non-goals

- Not an agent harness, planner, or chat UI (Claude Code/Claudian are the harness)
- Not a RAG/semantic-search product (integrate, don't compete; Basic Memory-style retrieval can sit beside it)
- Not a vault reorganizer — overlay, never migration
- No cloud sync of vault content in v1

## 4. Users

- **Primary:** Obsidian power users running Claude Code/Codex/Claudian against their vault
- **Secondary:** Developers wanting governed persistent memory for any local agent (vault-agnostic: works on any markdown folder, including `~/.claude` memory dirs)
- **Later:** Teams sharing a governed vault (paid tier)

## 5. Core concepts

### 5.1 Zones (permissions manifest — `.ledger/permissions.yaml`)

| Zone | Example | Agent rights |
|---|---|---|
| **Trusted** | user's own notes, `Projects/**` | read; writes require approval |
| **Agent** | `Agent/Memory/**`, `Agent/Sessions/**` | read/write via broker, auto-committed |
| **Scratch** | `Agent/Scratch/**` | read/write, relaxed rules, TTL auto-expiry |
| **Excluded** | `Private/**`, `.obsidian/**` | invisible to agent |

Defaults generated by a read-only onboarding scan; user confirms. Modes: **Safe** (agent zone only), **Assisted** (trusted-zone edits allowed with approval — default), **Autonomous** (per-folder auto-apply with Git rollback).

### 5.2 Memory lifecycle

`scratch → working → canonical`, with `retired` and `forgotten` as terminal states:

- **Scratch:** session observations; TTL (default 14 days) then auto-archived
- **Working:** survives sessions; promoted from scratch by rule (referenced ≥N times, or agent proposes with reason)
- **Canonical:** the agent's durable beliefs; promotion **always** requires human approval
- **Retired** *(v0.3)*: no longer current knowledge, but visible in history and queryable with `status: retired`. Represented as structured metadata (status flip + `retired_reason` + optional `superseded_by`) applied as a minimal patch — never appended prose. Distinct from **forgotten** (suppressed per policy; tombstoned to Archive).
- **Distilled memories** *(v0.3)*: a memory derived from other memories carries a `derivation` block naming its sources. If a source is later retired or revised, the distillation is flagged stale in the audit queue — derived knowledge is only as good as its inputs, and the ledger can prove it.
- Every memory carries provenance frontmatter:

```yaml
---
ledger:
  id: mem_8f3a
  status: working
  created: 2026-07-02T10:45:00Z
  source: claude-code/session-2026-07-02-a
  reason: "User confirmed project deadline moved to Aug 15"
  confidence: high        # enum authoritative; optional agent-supplied score recorded but never gating
  supersedes: mem_71bc
  expires: null
  derivation:             # v0.3; omitted for direct memories
    kind: distilled
    sources: [mem_71bc, mem_42aa]
---
```

Agent-supplied numeric confidence is recorded as evidence, never authority: approval status and source type gate lifecycle transitions, not the model's self-score.

### 5.3 Transactions

Agents emit structured operations, never raw writes:

```json
{
  "op": "revise",
  "path": "Agent/Memory/Projects/Nova.md",
  "expected_hash": "sha256:abc123",
  "patch": "...unified diff...",
  "reason": "Client renamed the deliverable",
  "session": "claude-code/2026-07-02-a"
}
```

Broker pipeline: validate path/zone → verify hash → lint patch (markdown syntax preserved: wikilinks, frontmatter, callouts, block refs untouched outside patch hunks) → apply → Git commit (`ledger: revise Nova.md [mem_8f3a] session-a`) → record in SQLite journal. Any failure = clean rejection with machine-readable reason the agent can react to. Rollback = `ledger undo <txn|session>` (Git revert).

### 5.4 Conflict & staleness detection

- **Write-time contradiction check:** new claim vs existing canonical notes on same entity (heading/frontmatter/embedding similarity) → write proceeds to *working* but a conflict item is queued; canonical is never silently contradicted
- **Staleness:** TTL sweep + "not referenced in N days" flags surfaced in the review queue
- Detection is heuristic in v1 (string/entity match), embedding-assisted in v1.1

## 6. Components

```
vaultledger/
├── packages/
│   ├── core/          # broker: zones, hashing, patch apply, Git, journal (SQLite)
│   ├── mcp-server/    # MCP tools: remember/recall/revise/promote/forget/status
│   ├── cli/           # ledger init|scan|approve|undo|log|status
│   └── obsidian-plugin/  # review queue, diffs, provenance hover, rollback UI
```

- **State separation:** `.ledger/` in-vault holds only `permissions.yaml` + config; SQLite journal/index lives in app-support dir keyed by vault ID (sync-safe)
- **Onboarding:** `ledger init <vault>` → read-only scan → vault profile + proposed zones → user approves → done. Nothing written to user notes.
- **Recall:** v1 = journal-indexed structured recall (by entity, tag, recency, status) — not semantic search. MCP `recall` returns memories with provenance so the agent can cite them.

## 7. MCP tool surface (v1)

| Tool | Purpose |
|---|---|
| `memory_recall(query, filters)` | retrieve memories + provenance |
| `memory_remember(content, entity, reason)` | create scratch/working memory |
| `memory_revise(id, patch, reason)` | patch existing memory |
| `memory_promote(id, target_status, reason)` | request lifecycle promotion |
| `memory_forget(id, reason)` | tombstone (never hard-delete; Git keeps history) |
| `memory_distill(content, sources, reason)` *(v0.3)* | create derived memory citing source memory ids; broker records the relations |
| `memory_retire(id, reason, superseded_by?)` *(v0.3)* | mark memory no longer current; minimal metadata patch, stays in history |
| `vault_propose_edit(path, patch, reason)` | propose edit to trusted-zone note (→ approval queue) |
| `ledger_status()` | zones, pending approvals, recent transactions |

## 8. Milestones

- **v0.1 (prove the loop):** core broker + MCP server + CLI approve/undo. Demo: Claude Code session writes memory, laptop reboot, next session recalls with provenance; bad write rolled back with one command.
- **v0.2 (the demo that sells):** Obsidian plugin — approval queue, diff view, rollback, provenance hover.
- **v0.3 (lifecycle + audit — absorbs Undertow concepts):** `memory_distill` + `memory_retire` + source-relations table; promotion rules; contradiction queue; source-linked staleness (distillation flagged when a source is retired/revised); memory-health report surfaced in plugin queues and `ledger memory audit`. Split into two cycles:
  - **v0.3a (this cycle):** contradiction queue + three v0.2-backlog hardening items — see `docs/design/specs/2026-07-05-vaultledger-v03a-design.md`. The contradiction matcher is built lineage-aware so it directly feeds v0.3b's source-linked staleness.
  - **v0.3b (Undertow merge, build-prompts Prompt 10.5)** — split into two sub-cycles:
    - **v0.3b-1 (SHIPPED):** the two lifecycle ops + data model — `memory_distill` (+ `memory_relations` table, rebuildable), `memory_retire` (gated status→retired with `superseded_by` validation), `retired` status, `derivation`/`retired_reason`/`superseded_by`/`score` provenance, recall exclusion, confidence-as-evidence. See `docs/design/specs/2026-07-08-vaultledger-v03b1-lifecycle-ops-design.md`.
    - **v0.3b-2 (next):** source-linked staleness (a `stale-source` conflict kind + kind-aware liveness filter) + `ledger memory audit` (state-based scan). Forward-pins P1/P2/P3 in the b-1 design doc.
- **v1.0:** polish, packaged installers, docs, second-brain/Claudian integration guides.
- **Later:** embeddings-assisted conflict detection, replay/regret analysis (which memories fed which answers, and which fed errors), team tier (shared vault, multi-agent write arbitration), audit export.

**Sequencing note (July 3, 2026):** the three v0.1 security fixes (`.ledger/**` always-excluded, case-fold zone resolution, realpath containment) are blockers and land before any v0.2/v0.3 work.

## 9. Success criteria

- A non-developer Obsidian user can point it at an existing vault and have Claude Code remembering across sessions in <10 minutes, with zero existing notes modified
- Every agent-caused change in the vault is answerable: *who wrote this, when, why, and how do I undo it*
- Existing harness/skill projects integrate rather than compete (their prompts call our tools)

## 10. Key risks

| Risk | Mitigation |
|---|---|
| Platform ships native governed memory | Local-first + user-owned files + harness-agnostic MCP |
| Approval fatigue | Zone defaults biased to autonomous-with-rollback; approvals only for trusted/canonical |
| Sync conflicts vs hash checks | Broker detects & re-bases or queues; design conflict UX in v0.1, not later |
| Obsidian CLI/Bases format drift | Adopt kepano/obsidian-skills conventions; patch-based edits minimize format exposure |
